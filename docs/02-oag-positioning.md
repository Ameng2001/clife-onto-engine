# OAG 定位纠偏卡 —— 本体语义层的本质是 OAG，不是 (Graph-)RAG

> 用途：团队与客户沟通时的"纠偏卡"。建设方案（给蒙草的《问草大模型》方案）为贴近客户认知，
> 反复用了 "Graph-RAG 约束求解防幻觉" 的话术；按方法论的深层主张，这把 **OAG 的功劳记到了
> RAG 头上**，不精确。本文把分界讲准，并给出"对内/对外该怎么说"的对照。
>
> 依据：方法论 `07 OAG 与 RAG 的分界`、`08 把 Schema 变成能力清单`。

---

## 0. 一句话定位

**本体语义层的本质是 OAG（Ontology-Augmented Generation，行动增强，Palantir AIP 提出），不是 RAG/Graph-RAG（只读检索增强）。**

- RAG 增强"**生成内容的依据**"——只读，挡不住"写错"。
- OAG 增强"**行动的能力与边界**"——受治理的写：发现操作 → 选操作填参 → 引擎校验 → 提交/回滚 → 决策血统。

> 注意 OAG 出处是 **Palantir**（AIP 体系），不是 Anthropic——别记错。

---

## 1. RAG vs OAG 核心对比（方法论 07 原表）

| 维度 | RAG | OAG |
|---|---|---|
| 增强的对象 | 生成内容的依据 | 行动的能力与边界 |
| LLM 输入 | 检索到的文档片段 | 业务对象 + 可用操作 |
| LLM 输出 | 自然语言答案 | 结构化操作调用 |
| 对世界的影响 | 只读 | 写操作（受规则约束） |
| 出错后果 | 答案不准 | 状态被错误改变 |
| 追溯性 | 难以还原决策依据 | 完整决策血统 |

**关键澄清（07 原文）**：GraphRAG / 知识图谱增强检索 **仍属"读"的范畴**——帮模型理解实体关系，
但**不提供受治理的写回路径**。OAG 解决的是另一个问题：Agent 被授权写操作时，如何保证写得对、写得可追溯。

---

## 2. 本仓库的核心三件套 = OAG 的三层能力

OAG 落地需要三层能力（07 第五节），缺一不可。本仓库逐条对应：

| OAG 能力 | 回答的问题 | 本仓库实现 |
|---|---|---|
| **操作发现** | Agent 怎么知道能做什么、参数是什么 | `intent/manifest.py` 能力清单（Schema→可用 Action+参数） |
| **执行约束** | 违规操作怎么被实时拦截 | `kernel/action_engine.py`：guard → 内存变更 → 写后规则 → **确定性回滚** |
| **决策追溯** | 三个月后怎么回答"为什么这样做" | `trust/audit.py`：审计**快照**（非日志） |

OAG 链路（07 第三节）在本仓库的落点：

```
NL 口语
  → 意图编译器(intent/compiler.py)：在能力清单内选一个操作并填参（LLM 概率推理发生在"选择层"）
  → Action 引擎(kernel)：guard→变更→写后规则→提交/回滚（确定性约束发生在"执行层"）
  → 审计快照(trust)：决策血统
```

**概率推理在选择层，确定性约束在执行层** —— 这正是"LLM 负责理解，本体负责边界"，落到写操作上。

---

## 3. 防幻觉发生在哪 —— 执行层，不是检索层

把"防幻觉"归给 Graph-RAG 是本仓库要纠正的核心误述。真正的防线：

> **LLM 可以提议一个语法合法、置信 0.95 的操作，本体引擎照样确定性回滚。**

活证据是 grass / chili demo 的 **C 用例**（等价于 07 第六节"采购 Agent 选 Beta 被认证阈值拦回"）：

| 场景 | LLM 提议 | 引擎裁决 |
|---|---|---|
| grass | `出一地一方(species=紫花苜蓿)` conf 0.95 | `乡土合规` 写后 hard 违反 → **回滚**，结构化拒绝 |
| chili | `制定种植方案(variety=紫色甜椒)` | `品种适配` 写后 hard 违反 → **回滚**，结构化拒绝 |

GraphRAG 只读，根本没有写回路径，**挡不住"写错"**；防幻觉只能发生在受治理的执行层。

---

## 4. 那 OQL / NebulaGraph 算 Graph-RAG 吗 —— 不算

| 问题 | 回答 |
|---|---|
| 图库 + OQL 是什么？ | 本体**对象/关系/规则的存储与查询基底**（OAG 治理的"那个世界的状态"） |
| 它服务谁？ | ① OAG 的**执行约束**层：function-backed 规则在**写时**查图谱（乡土合规/品种适配读名录）；② 带成本计量、防注入的**受治理只读查询** |
| 是 Graph-RAG 管线吗？ | **不是**。没有 "embedding 切片 → 拼进 prompt → 生成" 那套；意图编译器接地用的是**能力清单 + 记忆**，不是图谱切片 |

一句话：**图谱是 OAG 的基底，OQL 是读这个基底的受控方式；不是 RAG 的替身。**

---

## 5. OAG 操作发现 vs Skill / MCP / CLI 工具发现

另一个比 Graph-RAG 更隐蔽的混淆：**"我接了 MCP / 配了一堆 Skill，所以我有 OAG 了。"**
不对。MCP/Skill/CLI 也提供"操作发现"，但发现的是**函数**；OAG 发现的是**一个共享受治理状态模型上的操作**。
清单都长得像（带参数的 callable 列表），区别全在调用周围、以及这些操作背不背靠同一个被规则治理的世界。

| 维度 | MCP / Skill / CLI 工具 | OAG Action |
|---|---|---|
| 能力从哪来 | **手写**的工具/技能清单（独立函数签名） | 从 **Schema 投影**（声明 writes 哪些对象、被哪些规则 gate）；改 schema/规则，清单自动变 |
| 治理在哪 | **在每个工具内部**，各写各的校验 → 散落、可被新工具绕过 | **引擎里、跨操作统一**：任何 Action 碰对象都过同一套写后规则，绕不过 |
| 事务性 | fire-and-forget，半成功无法回滚 | guard→变更→校验→**提交/回滚** 的事务契约 |
| 可追溯 | 顶多 log "调了工具X(参数Y)" | **决策血统**：状态快照 + 评估了哪些规则 + 裁决 |
| 操作之间 | 语义孤岛，靠 LLM 拿字符串粘 | 都作用于**同一对象图**，前一步写入立即对下一步 guard 可见 → 靠共享状态组合、多跳 |

**一句话**：一个 Skill/工具是"没有共享世界的动词"；一个 OAG Action 是"绑定在共享、受规则、可审计的世界上的动词"。

**层次澄清——MCP 是"线"，OAG 是"线背后的东西"**：二者不在一层、不是竞品。MCP/CLI 是把"可调用的东西"
暴露给 LLM 的**传输/协议**；OAG 是对本体做受治理写操作的**架构**。你完全可以把 OAG 的 Action 经 MCP 暴露
（本仓库 `intent/manifest.py` 能力清单就能渲染成一组 MCP tools），但每个"tool"背后坐着 Action 引擎
（guard/写后规则/回滚/审计），而不是裸函数。

**判断你是不是真有 OAG 的三问**（只有工具发现+调用 ≠ OAG）：
1. 这些操作**共享一个对象/规则模型**吗？（还是各自孤岛）
2. 有**集中、跨操作、写不进就回滚**的规则执法吗？（还是校验散在每个工具里）
3. 每次执行留**状态快照级的决策血统**吗？（还是只有调用日志）

三个都"是"→ OAG；否则只是"给 LLM 一个 API"——方法论明确说那不是 OAG。本仓库 `manifest.py` 看着像
"列可用 Skill"，但它从五要素 schema 投影、背后是 `kernel/action_engine.py` + `trust/audit.py`；辣椒/grass 的 C 用例
（LLM 选了合法操作、引擎照样回滚）就是纯工具/Skill 体系给不出来的行为——因为它们没有那个被规则治理的共享世界。

## 6. 纠偏卡：建设方案话术 → OAG 准确表述

对外沿用客户熟悉的词没问题，但**对内必须知道真相**；对外也建议逐步换成准确表述。

| 建设方案/常见话术（不精确） | OAG 准确表述（应这么理解/这么说） |
|---|---|
| "Graph-RAG 约束求解防幻觉是核心" | **OAG 执行层防幻觉**是核心：LLM 在能力清单内提议，引擎 guard/写后规则/回滚兜底 |
| "本体大模型 = 通用模型 + 草业 Graph-RAG" | 本体语义层 = **OAG**（操作发现+执行约束+决策追溯）；RAG 只是可选的只读补充 |
| "先图谱检索定位，再向量召回，喂给 LLM 生成方案" | 先用本体**约束出可行操作/参数空间**（OAG 选择层），引擎**执行并可回滚**（执行层）；图谱查询是约束，不是检索投喂 |
| "RAG 三通道（图谱/向量/混合）是决策引擎" | 三通道是**只读检索**，服务开放问答；**决策与执行**走 OAG（Action 引擎），二者分层、别混为一谈 |
| "知识查询必经知识库 = 防幻觉" | 知识查询（读）防的是"答错"；防"做错/写错"必须靠 OAG 的受治理写回路径 |
| "接了 MCP / 配了一堆 Skill 就是 OAG 了" | 那只是**工具发现+调用**；OAG 还要共享状态模型 + 集中规则执法 + 回滚 + 决策血统（见 §5） |

---

## 7. 不是要否定 RAG —— 分清各自的活

RAG / Graph-RAG **仍有正当用途**，只是别越位：

- **该用 RAG 的**：开放问答、政策法规摘录、文献总结、案例语义召回——这些是"读"，RAG 合适。
- **该用 OAG 的**：出方案、评级、签发凭证、派工单、申报补贴——这些是"受治理的写"，必须 OAG。
- **混合**：一次交互里可以先 RAG 读资料辅助理解，再走 OAG 执行——但**两层职责要分清**，不要把 RAG 说成决策/执行引擎。

---

## 8. 一句话记忆法

> **RAG 让 AI "会查"，OAG 让 AI "会做"。**
> 本体语义层是为了让 AI **会做且做得对、可追溯**——所以它是 OAG。
> 图谱是 OAG 治理的世界状态，不是把它退化成一个更大的检索库。
